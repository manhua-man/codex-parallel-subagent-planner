# 请求链路

本文说明 `parallel-subagent-planner` 在一次 Codex 请求中的位置。这个 skill 不是 runtime 组件、本地 runner、Python 包、CLI 或 session manager。它由 Codex 加载为指令，用来指导主线程判断是否拆分、是否启动子智能体，以及如何做最终集成。

```text
  |      |
  |      v
  |   parallel-subagent-planner 作为 skill 可用
  |
  +--> 恢复 session 状态
  |
  +--> 组装内置 tools
  |

  |
  +--> 不适用：主线程直接处理任务
  |
  +--> 适用：主线程按下面的 skill flow 执行
```

## 职责边界

Codex 负责：

- 加载 skill 到上下文
- 提供真实的子智能体启动能力
- 管理 session、tools、文件访问和终端执行
- 渲染最终回复

这个 skill 负责指导：

- 是否应该考虑子智能体
- 哪些 lane 足够安全、值得启动
- 哪些 lane 应该 hold、合并或留在主线程
- 每个子智能体拿到什么 compact context
- 每个子智能体可以读写哪些范围
- 子智能体结果如何集成和验证
- 什么时候询问用户是否沉淀可复用子智能体角色

## 强制长期 Agent 检查

每次 parallel run 结束后，主线程都要检查有没有已完成的子智能体角色值得沉淀成长期 agent。如果有可复用候选，主线程主动问用户是否要 promote。

输出只应该说明：

- 这个角色值不值得沉淀
- 为什么值得
- agent spec 建议存放在哪里

在用户明确批准具体 promotion 之前，不要创建或写入 `.codex/agents/` 或 `agents/` 文件。

## 非目标

本仓库不提供：

- Python 包
- CLI
- 本地 task runner
- session manager
- Codex 子智能体启动能力的替代品
- 未经明确批准自动创建 `.codex/agents/` 或 `agents/` 文件
